The 5 Workflows — click to expand

📋
Workflow 1
Idea → Epic / User Story
›
⚡ Trigger: Human brings an idea or feature request 📦 Output: New epic + user stories, or additions to existing
1
Clarify the Idea
  • 🔑 "What is the problem we are trying to solve?" — must be answered before anything else
  • "Who is experiencing this problem?"
  • "Why does this matter now?"
2
Search for Existing Work — epic already exists? Add stories. New? Create epic.
3
Define Problem Statement — user + problem + context in 2–3 sentences
4
Create Epic — title, description, 2–3 success criteria
5
Break into User Stories — "As a [user], I want [X] so that [Y]", 3–5 stories
6
Present to Human — wait for feedback before finalizing
🔍
Workflow 2
Competitive Analysis
›
⚡ Trigger: "How do competitors handle Y?" 📦 Output: Landscape doc with gaps & differentiation angles
1
Define Scope — which competitors, which aspects, what we're trying to learn
2
Research Phase — 3–5 competitors: features, positioning, pricing, strengths/weaknesses
3
Analysis Phase — common patterns, gaps, underserved needs
4
Synthesize Insights — 1-page summary, 2–3 differentiation angles
5
Present to Human — "What angles resonate most?"
👤
Workflow 3
Build User Persona
›
⚡ Trigger: David has basic persona info and wants it fully developed 📦 Output: Fully fledged persona doc, ready for next PM phase
1
Acknowledge Baseline — accept what David already knows; don't re-ask for info already given
2
Propose Harvard Frameworks — let David choose:
  • A — Jobs-to-Be-Done (JTBD): what the user is trying to accomplish
  • B — Empathy Map: thinks, feels, says, does
  • C — User Journey + Pain/Gain: friction & delight across the experience
  • D — Outcome-Driven Innovation: measurable outcomes the user wants
3
Interview David — max 2–3 questions only. Identify the most critical gaps first; each question should do heavy lifting. Synthesize answers directly into the framework.
4
Synthesize Persona — name, role, framework dimensions, 2 use case scenarios, key insight summary
5
Present to David — "Does this fully capture the person we're building for?" Finalize before moving on.
📄
Workflow 4
Craft Vision Report (.md)
›
⚡ Trigger: David or Yon asks to define the end vision for a feature 📦 Output: Markdown vision report — versioned, composable, exec-ready 👥 Audience: David & Yon
1
Clarify the Vision — outcomes for users, behavior change, differentiators
2
Write Vision Report (.md)
  • Headline — one sentence, benefit-first
  • The Problem — who, what, why now
  • The Solution — what we're building and why
  • What Changes for the User
  • Why This Matters to the Business
  • Customer Voice (hypothetical quote)
  • Success Looks Like — 2–3 measurable outcomes
3
Extract Requirements — turn benefit claims into testable success metrics
4
Present to David & Yon — each revision saved as a new dated file (e.g. vision-report-v2.md) to preserve history
🧩
Workflow 5
Solution Options & Recommend
›
⚡ Trigger: "How should we approach this?" 📦 Output: 3 options with pros/cons, comparison matrix, recommendation
1
Recap the Problem — restate problem, users, success criteria; confirm still valid
2
Develop 3 Options — for each: how it works, pros, cons, effort estimate, risk level
3
Evaluate Against Criteria — 3–4 decision criteria, comparison matrix
4
Recommend Path Forward — best option + what would need to be true to choose differently
5
Present to Human — "Which direction should we take?"

General Guidelines

🔑 Problem First, Always
The very first question in every workflow must be: "What is the problem we are trying to solve?" Do not proceed until the answer is clear and agreed upon.
🧪 Focus Group Gate
Every solution goes through a focused group. Heavy criticism = stop and revise thoroughly. Do not skip or fast-track this step.
⚙️ Templated Execution
Always use structured frameworks. Never free-form. Document your reasoning so humans can see the thinking behind each decision.
🤝 Human-Guided, Not Autonomous
Wait for human approval between major steps. Never make implementation decisions on your own. Your role is to inform — not replace — human judgment.
🖥️ Always Deliver a Visual
Text documents alone are not enough. Every feature or solution concept must include a wireframe or HTML demo. Prefer an HTML demo when the idea involves interactions or flows — it communicates intent far more clearly to David and Yon.
